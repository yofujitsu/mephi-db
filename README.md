## Входные данные и условия

### БД 1. Транспортные средства

**Таблицы:**
- `Vehicle` — базовая таблица с производителем, моделью и типом
- `Car` — характеристики автомобилей (объём двигателя, мощность, цена, трансмиссия)
- `Motorcycle` — характеристики мотоциклов (объём двигателя, мощность, цена, тип)
- `Bicycle` — характеристики велосипедов (количество передач, цена, тип)

**Задачи:**
- Задача 1: Запрос мотоциклов с заданными критериями (мощность > 150 л.с., цена < 20000, тип Sport)
- Задача 2: Комбинированный запрос транспортных средств с различными критериями с использованием UNION

### БД 2. Автомобильные гонки

**Таблицы:**
- `Classes` — классы автомобилей с типом, страной, спецификациями
- `Cars` — названия автомобилей, привязанные к классам
- `Races` — названия гонок и даты
- `Results` — результаты гонок, связывающие автомобили с гонками и позициями

**Задачи:**
- Задача 1: Найти автомобили с наилучшей средней позицией в каждом классе
- Задача 2: Найти автомобиль с наилучшей общей средней позицией
- Задача 3: Найти классы с наилучшей средней позицией и общим количеством гонок
- Задача 4: Найти автомобили со средней позицией лучше средней по классу
- Задача 5: Найти классы с наибольшим количеством автомобилей с низкой средней позицией

### БД 3. Бронирование отелей 

**Таблицы:**
- `Hotel` — информация об отелях (название, расположение)
- `Room` — детали номеров (тип, цена, вместимость)
- `Customer` — информация о клиентах (имя, email, телефон)
- `Booking` — записи о бронированиях с датами заезда и выезда

**Задачи:**
- Задача 1: Найти клиентов, сделавших более двух бронирований в разных отелях
- Задача 2: Анализ клиентов с > 2 бронированиями и суммой трат > 500
- Задача 3: Категоризация отелей по цене и определение предпочтений клиентов

### БД 4. Структура организации

**Таблицы:**
- `Departments` — названия отделов
- `Roles` — названия ролей
- `Employees` — иерархия сотрудников с отношениями подчинения
- `Projects` — проекты, привязанные к отделам
- `Tasks` — задачи, назначенные сотрудникам

**Задачи:**
- Задача 1: Рекурсивный запрос для нахождения всех сотрудников, подчиняющихся генеральному директору (Иван Иванов)
- Задача 2: Рекурсивный запрос с подсчётом задач и подчинённых
- Задача 3: Найти менеджеров с подчинёнными с использованием RECURSIVE

## Как запустить проект

### Запуск с Docker Compose

1. Запустить контейнер postgres со монтированными базами данных и решениями:
   ```bash
   docker-compose up -d
   ```

2. Проверить, что контейнер запущен:
   ```bash
   docker ps
   ```
   Должен быть виден контейнер `mephi-db-postgres`.

3. Подключиться к БД через терминал:
   ```bash
   docker exec -it mephi-db-postgres psql -U postgres
   ```

4. Проверить, что таблицы подтянулись:
   ```sql
   \dt
   ```

### Как протестировать решения

#### Через терминал:

Все файлы решений смонтированы в контейнере в директории `/solutions/`. Выполнение данных sql-скриптов выводит ответы сразу на все задачи к конкретной БД:

```bash
# База данных 1: Транспортные средства
docker exec -i mephi-db-postgres psql -U postgres -f /solutions/01_vehicles_solutions.sql

# База данных 2: Автомобильные гонки
docker exec -i mephi-db-postgres psql -U postgres -f /solutions/02_racing_solutions.sql

# База данных 3: Бронирование отелей
docker exec -i mephi-db-postgres psql -U postgres -f /solutions/03_hotels_solutions.sql

# База данных 4: Структура организации
docker exec -i mephi-db-postgres psql -U postgres -f /solutions/04_organization_solutions.sql
```

#### Через pgAdmin

1. Открыть pgAdmin
2. Создать новое соединение:
   - **Host:** `localhost`
   - **Port:** `5432`
   - **Username:** `postgres`
   - **Password:** `postgres`
   - **Database:** `postgres`
3. Далее можно копипастить sql-запросы из файлов решений в UI

#### Если нужен перезапуск

Для полного сброса данных (удаления volume):

```bash
docker-compose down -v
docker-compose up -d
```

## Известные расхождения

### База данных 2, Задача 5

**Описание проблемы:** Ожидаемый вывод в задании содержит 4 класса (Sedan, Coupe, Hatchback, Pickup), однако при использовании условия `>= 3.0` для фильтрации автомобилей с низкой средней позицией, класс SUV также должен присутствовать в выводе (Nissan Rogue имеет позицию 3.0).

**Обоснование решения:** В решении используется условие `>= 3.0` (вместо `> 3.0`), так как:
- При `> 3.0` класс Sedan имел бы count = 1 (только Audi A4), но ожидаемый вывод показывает count = 2
- При `>= 3.0` класс Sedan имеет count = 2 (BMW 3 Series = 3.0, Audi A4 = 8.0), что соответствует ожидаемому выводу
- Класс SUV (Nissan Rogue = 3.0) также удовлетворяет условию `>= 3.0`, но отсутствует в ожидаемом выводе

**Вывод:** Решение корректно использует `>= 3.0` и возвращает 5 классов (включая SUV). Ожидаемый вывод в задании, вероятно, содержит неточность, исключая SUV из результатов.

## Какие функции SQL были задействованы

- **Операции JOIN**: INNER JOIN, LEFT JOIN
- **Операции над множествами**: UNION, UNION ALL
- **Агрегатные функции**: COUNT, SUM, AVG, MIN, MAX
- **Оконные функции**: SUM() OVER (PARTITION BY ...)
- **Обобщённые табличные выражения (CTE)**: WITH
- **Рекурсивные CTE**: WITH RECURSIVE для иерархических запросов
- **Агрегация строк**: STRING_AGG для конкатенации значений
- **Условная логика**: CASE WHEN
- **Фильтрация**: HAVING для группированных данных
